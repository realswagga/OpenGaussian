# Rendering Pipeline Optimization Plan

## 1. Current Architecture Analysis

### 1.1 Rendering Pipeline (Line-by-Line)

The codebase has **two rendering paths**:

| Path | Entry Point | Status |
|------|-------------|--------|
| **Primary** | `PlayCanvasGsplatRuntime.ts` → PlayCanvas Engine `gsplat` component | Active, production |
| **Legacy/Fallback** | `InstancedQuadRenderer.ts` → raw WebGL2 instanced quads | In code but not wired into production flow |

**Primary path flow** (`PlayCanvasGsplatRuntime.ts`):
1. `start()` → `createApp()` (lines 431–457): Creates PlayCanvas `AppBase` with `GSplatComponentSystem`, selects WebGL2 or WebGPU
2. `loadGsplatAsset()` (lines 509–569): Loads asset via PlayCanvas `GSplatHandler`, creates `gsplat` component with `unified: true`
3. `applyGsplatSettings()` (lines 584–595): Sets `splatBudget`, `antiAlias`, `alphaClip=1/255`, renderer mode (`RASTER_CPU_SORT` or `RASTER_GPU_SORT`)
4. Per-frame: PlayCanvas engine internally sorts (CPU radix via `gsplat-sorter.js` worker or GPU radix), then rasterizes instanced quads
5. `resize()` (lines 632–644): Caps pixel ratio via `maxDevicePixelRatio`, clamps canvas to `maxPixelDim` (1080 mobile / 2160 desktop)

**Legacy path flow** (`InstancedQuadRenderer.ts`):
1. `upload()` (lines 355–403): Creates 6 VBOs (pos, scale, rot, color, opacity, index) with `STATIC_DRAW`
2. Per-frame: `RadixSorter.sortDepths()` computes dist², radix-4-pass, reverses for back-to-front
3. `updateSortIndices()` (line 432–437): `bufferSubData` on index VBO
4. `draw()` (lines 454–495): Single `drawArraysInstanced(TRIANGLE_STRIP, 0, 4, capacity)` with premultiplied alpha blend

### 1.2 Shader Optimizations (Current)

**Vertex shader** (`InstancedQuadRenderer.ts:27–158`):
- Full quaternion→3×3 rotation matrix computation per vertex (9 multiplies, 9 additions)
- Full 3×3 world covariance Σ₃ = R·diag(s²)·Rᵀ (6 unique entries, ~27 ops)
- Projection Jacobian J (2×3) computation + Σ₂ = J·Σ₃·Jᵀ (~20 ops)
- Eigenvalue clamping + 3σ quad expansion
- **No optimizations**: no precomputed covariance, no half-precision option, no early culling

**Fragment shader** (`InstancedQuadRenderer.ts:161–192`):
- Mahalanobis distance² computation per fragment
- Gaussian exp(-0.5·d²) per fragment
- Alpha clip at 1/255 threshold
- **No optimizations**: no bounding-box early discard, no `mediump` mobile variant, no 2σ vs 3σ tradeoff

**PlayCanvas engine shaders** (internal, not in this repo):
- PlayCanvas v2.18+ uses its own GSplat vertex/fragment programs with built-in sort integration
- Supports `highPrecisionRendering` flag (SuperSplat `?hpr`)
- Uses WebGPU compute shaders for GPU radix sort when `webgpu` renderer selected

### 1.3 Memory Allocation Strategies

| Component | Allocation | Strategy | Issues |
|-----------|------------|----------|--------|
| InstancedQuadRenderer VBOs | `gl.bufferData(STATIC_DRAW)` | One-shot upload, never resized | Index VBO updated every frame but marked STATIC |
| InstancedQuadRenderer capacity | `this.capacity = N` at upload time | Fixed, no growth | Must destroy/recreate if splat count changes |
| RadixSorter buffers | `bufA/bufB = Uint32Array(N*2)` at construction | Pre-allocated, `resize()` reallocates | No shrink strategy; `resize()` only grows |
| RadixSorter histogram | `Uint32Array(256)` | Fixed 256 bins, reused | Efficient, no issues |
| PlayCanvas gsplat | Internal to engine | Streaming LOD with `frame:ready` events | Budget caps via `splatBudget` property |
| LOD chunks (worker) | `cell.vertices.length * vertexSize` per chunk | Per-cell allocation, no pooling | Each chunk is a full PLY binary (62 floats × 4 bytes × vertexCount) |

**Per-splat memory footprint**:
- PLY format: 62 floats × 4 bytes = **248 bytes/splat**
- SOG format: **40 bytes/splat** (compressed)
- SPZ format: **~60 bytes/splat**
- InstancedQuadRenderer upload: pos(12) + scale(12) + rot(16) + color(12) + opacity(4) + index(4) = **60 bytes/splat GPU-side** (excludes SH, excludes covariance)

### 1.4 Draw Call Management

**Current state**: **1 draw call per frame** for all visible splats.

- `InstancedQuadRenderer.draw()`: `gl.drawArraysInstanced(TRIANGLE_STRIP, 0, 4, this.capacity)` — renders ALL uploaded splats regardless of visibility
- PlayCanvas gsplat: single draw call bounded by `splatBudget` (engine caps instance count)
- **No frustum culling**: off-screen splats are sorted and submitted
- **No screen-size culling**: sub-pixel splats are fully rasterized
- **No opacity culling**: near-transparent splats below alphaClip are only discarded in fragment shader (still sorted and vertex-processed)

### 1.5 Culling Mechanisms

**Currently: ZERO explicit culling in this codebase.**

- No frustum culling (no intersection test against camera frustum)
- No distance culling (no max-range discard)
- No screen-size culling (no projected-size threshold)
- No occlusion culling (no hidden-surface detection)
- No LOD distance-based selection in runtime code (PlayCanvas engine handles LOD streaming internally via `frame:ready`, but no distance thresholds are specified in the manifest)
- The only "culling" is:
  - `splatBudget` — caps total rendered count, but selects by sort priority (nearest first), not by visibility
  - `alphaClip = 1/255` — fragment discard for nearly-transparent splats
  - Marker visibility: `updateMarkerPositions()` (line 956) clips markers at canvas bounds ±64px

### 1.6 LOD System (Current)

**Worker-side generation** (`apps/worker/src/index.ts:592–806`):
- Spatial grid: 4×4×4 = 64 cells
- LOD levels: 4 tiers with **arbitrary sampling ratios**:
  - `lod-0`: 100% (full)
  - `lod-25`: 25%
  - `lod-6`: 6%
  - `lod-1-5`: 1.5%
- Sampling: **stride-based** (evenly spaced indices), NOT importance-weighted
- Minimum chunk size: 10 vertices (line 727)
- Output: full 62-float PLY per chunk (no compression, no SOG conversion)
- Manifest: `lod-meta.json` with bounds per cell but **NO transition distances**

**Runtime LOD** (`PlayCanvasGsplatRuntime.ts`):
- `resolvedAsset.isLod` determines LOD path
- `frame:ready` events with `loadingCount` track streaming progress
- `lodLoadingCount` tracked, progress phases: `refining` → `complete`
- **No runtime distance-based LOD selection visible** — PlayCanvas engine's internal `gsplat-instance.js` handles this, but the manifest doesn't specify transition thresholds

## 3. Structural Inefficiencies: Why `phoneLow` Is Too Heavy

### 3.1 Mathematical Analysis of phoneLow at 400K Splats

**GPU memory (raw PLY upload)**:
- 400K splats × 248 bytes (PLY) = **99.2 MB** just for source data
- 400K × 60 bytes (GPU-side: pos+scale+rot+color+opacity+index) = **24 MB** GPU buffers
- Total GPU allocation: ~24 MB (acceptable for low-end)

**But the real bottleneck is FRAGMENT throughput**:

On a low-end phone (e.g., Snapdragon 680, Adreno 610):
- Fill rate: ~400M pixels/sec at best
- At DPR=1.0, 1080×2340 canvas = **2.53M pixels**
- Average splat screen coverage at medium distance: ~200 pixels/splat (3σ ellipse)
- With alpha blending: each pixel may blend 5–15 overlapping splats
- **Fragment throughput needed**: 400K × 200 avg_pixels × 10 avg_overlaps = **800M fragment ops/frame**
- At 400M pix/sec: **~2 seconds per frame** → 0.5 FPS

Even with conservative estimates (50 pixels/splat average, 3 overlaps):
- 400K × 50 × 3 = **60M fragments** → 60M/400M = ~150ms → **6.7 FPS**

**Sort bottleneck on low-end CPU**:
- RadixSorter: 400K × 4 passes × (histogram + scatter) ≈ 400K × 4 × ~8 ops = ~12.8M ops
- Depth² computation: 400K × 3 multiplies + 2 adds = ~2M ops
- On a ~2 GFLOPS mobile CPU core: sort alone ≈ **8–15ms**
- PlayCanvas worker-based sort adds message-passing overhead: ~5ms
- Total sort: **15–20ms per frame** — leaves only 13ms for rendering at 30 FPS target

### 3.2 Why 400K Is Still Excessive

For true low-end viability (≥30 FPS on Adreno 610-class):
- Frame budget: 33ms (30 FPS)
- Sort budget: ≤10ms → max ~150K splats
- Render budget: ≤23ms → max ~150K visible fragments at ≤30 avg_pixels
- **Target: 150K splatBudget for true low-end**, not 400K

For VR (72 FPS, Quest 2, Adreno 650):
- Frame budget: 13.9ms (72 FPS)
- Sort budget: ≤5ms → max ~200K splats with GPU sort
- Render budget: ≤9ms → at DPR=1.0, dual-eye 1832×1920 = ~3.5M pixels, max ~200K splats
- **Target: 200K splatBudget for true Quest low-end**, not 700K

#### 4.P0.2: Convert LOD chunks from PLY to SOG format

**Current**: Worker outputs full 62-float PLY chunks (248 bytes/splat)
**Proposed**: Output SOG-compressed chunks (40 bytes/splat) — **6.2× bandwidth reduction**

Impact on 150K phoneLow budget:
- PLY download: 150K × 248B = 37.2 MB
- SOG download: 150K × 40B = 6 MB
- **Network load time reduced by 83%**

**Files to change**:
- `apps/worker/src/index.ts` — LOD generation: convert chunk output to SOG format instead of raw PLY
- This requires PlayCanvas's SOG compression in the worker pipeline (may need a conversion library or call PlayCanvas's `gsplat-sog-data.js` logic)

#### 4.P0.3: Add frustum culling before sort

**Current**: All splats within `splatBudget` are sorted and rendered
**Proposed**: Before sort, filter splats whose center is outside the camera frustum ± margin

Implementation:
- Compute 6 frustum planes from `camera.projectionMatrix × camera.matrixWorldInverse`
- For each splat position, test against planes (6 dot products per splat)
- Mark off-screen splats as "skip" in sort input
- Expected culling rate: 30–60% of splats at typical viewing angles
- On 150K phoneLow: cull 45K–90K → effective sort/render = 60K–105K → **2× FPS improvement**

**Files to change**:
- `packages/viewer-core/src/PlayCanvasGsplatRuntime.ts` — add frustum culling before PlayCanvas sort, or configure PlayCanvas engine's built-in culling (investigate if engine supports this natively)
- `packages/viewer-core/src/render/InstancedQuadRenderer.ts` — add frustum culling to draw path
- `packages/viewer-core/src/sort/RadixSorter.ts` — accept a visibility mask to skip off-screen splats

#### 4.P0.4: Add screen-size culling (sub-pixel splat discard)

**Current**: All in-budget splats are rendered regardless of projected size
**Proposed**: Discard splats whose projected 3σ radius < 1.5 pixels

Implementation:
- In vertex shader (or pre-sort): compute `radius = 3.0 * sqrt(eigMax)`
- If `radius < 1.5 pixels`, mark splat as "skip"
- On typical scene: 10–20% of splats are sub-pixel at medium distance
- Saves fragment shader invocations with zero visual impact

**Files to change**:
- `packages/viewer-core/src/render/InstancedQuadRenderer.ts` — add size threshold uniform and early discard in vertex shader
- `packages/viewer-core/src/PlayCanvasGsplatRuntime.ts` — configure PlayCanvas engine's `minSplatSize` if available, or add custom pre-filter

### P1 — High Impact (Significant frame-rate improvement, moderate complexity)

#### 4.P1.1: Add distance-based LOD transition thresholds to manifest

**Current**: LOD manifest has no transition distances; ratios are arbitrary (100%/25%/6%/1.5%)
**Proposed**: Add `transitionDistances` array to `lod-meta.json`, calculated from scene bounding box

Mathematical derivation:
- For a scene with bounding sphere radius `R` and camera at distance `d`:
- Projected splat coverage ≈ `splatSize² / d² × focalLength²`
- LOD transition when projected coverage drops below threshold:
  - LOD-0 → LOD-25: transition at `d = 0.3 × R` (close-up still needs detail)
  - LOD-25 → LOD-6: transition at `d = 1.0 × R` (medium distance)
  - LOD-6 → LOD-1.5: transition at `d = 3.0 × R` (far distance)
- These thresholds ensure perceptual quality matches distance

**Proposed LOD levels (recalibrated)**:

| LOD | Ratio | Name | Transition Distance | Justification |
|-----|-------|------|---------------------|---------------|
| lod-0 | 100% | lod-full | 0 → 0.3R | Full detail at close range |
| lod-25 | 25% | lod-quarter | 0.3R → 1.0R | Quarter detail at medium range (75% reduction acceptable at 3× distance) |
| lod-6 | 6% | lod-sparse | 1.0R → 3.0R | Sparse for far range (94% reduction at 9× projected size reduction) |
| lod-1-5 | 1.5% | lod-hull | 3.0R → ∞ | Bare silhouette for very far (1.5% sufficient when projected < 5px) |

**Files to change**:
- `apps/worker/src/index.ts` — compute `transitionDistances` from bounding sphere radius and include in `lod-meta.json`
- `packages/viewer-core/src/types.ts` — add `transitionDistances` to manifest type
- `packages/viewer-core/src/PlayCanvasGsplatRuntime.ts` — pass transition distances to PlayCanvas gsplat instance
- `packages/viewer-core/src/gsplatCenters.ts` — parse transition distances from manifest

#### 4.P1.2: Add adaptive quality (dynamic splatBudget based on frame timing)

**Current**: Fixed `splatBudget` per profile, no runtime adjustment
**Proposed**: Measure frame time over last 8 frames, adjust `splatBudget` dynamically

Algorithm:
- Target FPS: phoneLow=30, phoneHigh=45, vrQuest=72, desktop=60
- If avg frame time > target+5ms: reduce budget by 10% (minimum: 50K)
- If avg frame time < target-5ms: increase budget by 5% (maximum: profile.splatBudget)
- Rate limit: adjust once per second, not per frame
- This ensures low-end devices self-tune to their actual capability

**Files to change**:
- `packages/viewer-core/src/PlayCanvasGsplatRuntime.ts` — add `adaptiveBudget` logic in `onUpdate`, modify `gsplat.splatBudget` dynamically
- `packages/viewer-core/src/types.ts` — add `targetFps` and `adaptiveBudgetEnabled` to `QualityProfile`
- `packages/shared/src/types.ts` — same additions
- `packages/shared/src/constants.ts` — same additions

#### 4.P1.3: Use `mediump` shader variant for mobile

**Current**: Both shaders declare `precision highp float`
**Proposed**: For phoneLow/phoneHigh profiles, compile a `mediump` variant

Impact:
- `mediump` on mobile GPUs: ~2× faster ALU, ~30% less register pressure
- Visual quality: negligible difference for splat rendering (colors are 8-bit anyway)
- Covariance computation in mediump: acceptable with eigenvalue clamping (already present)

**Files to change**:
- `packages/viewer-core/src/render/InstancedQuadRenderer.ts` — add `MEDIAN_PRECISION` flag, generate `mediump` shader variant
- `packages/viewer-core/src/PlayCanvasGsplatRuntime.ts` — select shader precision based on profile

#### 4.P1.4: Reduce quad expansion from 3σ to 2σ for mobile

**Current**: `radius = 3.0 * sqrt(eigMax)` (covers 99.7% of Gaussian)
**Proposed**: For mobile profiles, use `radius = 2.0 * sqrt(eigMax)` (covers 95.4%)

Impact:
- 2σ quad is 44% smaller than 3σ quad → **44% fewer fragment shader invocations**
- Visual quality: only 4.6% of Gaussian energy is outside 2σ, nearly invisible difference
- Alpha at 2σ boundary: exp(-2) = 0.135, still visible; at 3σ: exp(-4.5) = 0.011, barely visible
- **This is the single highest-impact shader optimization for mobile**

**Files to change**:
- `packages/viewer-core/src/render/InstancedQuadRenderer.ts` — add `uSigmaScale` uniform (2.0 or 3.0), replace hardcoded `3.0` with uniform
- For PlayCanvas path: investigate if engine supports `sigmaScale` configuration

### P2 — Medium Impact (Improves memory/stability, lower urgency)

#### 4.P2.1: Change index VBO from STATIC_DRAW to DYNAMIC_DRAW

**Current**: `gl.bufferData(gl.ARRAY_BUFFER, data, gl.STATIC_DRAW)` for index VBO
**Proposed**: Use `DYNAMIC_DRAW` for the index buffer that's updated per-frame

**Files to change**:
- `packages/viewer-core/src/render/InstancedQuadRenderer.ts:397-400` — separate index VBO creation with DYNAMIC_DRAW hint

#### 4.P2.2: Increase LOD spatial grid resolution

**Current**: 4×4×4 = 64 cells
**Proposed**: 8×8×8 = 512 cells

Impact:
- Better spatial granularity → fewer irrelevant chunks loaded
- Smaller per-chunk memory footprint → better LOD streaming
- Trade-off: more manifest entries, more HTTP requests → mitigated by chunk batching

**Files to change**:
- `apps/worker/src/index.ts` — change `gridSize` from 4 to 8, update cell computation

#### 4.P2.3: Add importance-weighted sampling for LOD generation

**Current**: Stride-based sampling (evenly spaced indices)
**Proposed**: Weight by splat opacity × projected size (larger, more opaque splats are more important)

Algorithm:
- For each cell, compute `weight[i] = opacity[i] × max(scale[i])`
- Sort vertices by weight descending
- Sample top-N by weight for each LOD level
- This preserves visually important splats in low-LOD levels

**Files to change**:
- `apps/worker/src/index.ts` — replace stride sampling with importance-weighted sampling in LOD generation

#### 4.P2.4: Add `?budget=` URL parameter override (SuperSplat parity)

**Current**: Budgets are hardcoded in profiles and server defaults
**Proposed**: Accept `budget` query parameter to override `splatBudget` dynamically

**Files to change**:
- `packages/viewer-widget/src/gs-viewer.ts` — parse `budget` attribute
- `packages/viewer-core/src/types.ts` — add `budgetOverride` to ViewerOptions
- `packages/viewer-core/src/PlayCanvasGsplatRuntime.ts` — apply budget override

#### 4.P2.5: Add `?nofx` URL parameter for post-effect toggle (SuperSplat parity)

**Files to change**:
- `packages/viewer-widget/src/gs-viewer.ts` — parse `nofx` attribute
- Wire to `enablePostFx` in profile override

### P3 — Lower Impact / Future Work

#### 4.P3.1: Precompute covariance on GPU (compute shader)

- Move quaternion→covariance computation to a WebGPU compute pass
- Store precomputed 2D covariance in a texture/buffer
- Eliminates per-vertex ALU in render pass
- Requires WebGPU path only

#### 4.P3.2: GPU radix sort for legacy renderer

- `InstancedQuadRenderer.ts:291-293` already has placeholder for transform feedback
- Implement GPU sort via WebGL2 transform feedback or WebGPU compute
- Eliminates CPU sort bottleneck entirely

#### 4.P3.3: Compressed texture format for splat data

- Pack (scale, rotation, color, opacity) into half-float or packed formats
- Reduce per-instance attribute buffer from 60 bytes to ~32 bytes
- Requires shader unpacking logic

#### 4.P3.4: Occlusion culling via depth pyramid

- Build hierarchical depth buffer from previous frame
- Test splat centers against depth pyramid before sort
- Expected additional culling: 10–30% of splats hidden behind foreground
- Complex implementation, deferred
