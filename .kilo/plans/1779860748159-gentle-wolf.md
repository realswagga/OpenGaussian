## Fix 2: WASD in Orbit Mode — Position vs Rotation Bug

**Problem**: In `PlayCanvasGsplatRuntime.ts:895-920` (`updateOrbitPan`), WASD projects `cam.forward` onto the horizontal plane (`forward.y = 0; forward.normalize()`). When the camera pitch is near ±90° (looking straight up/down), the horizontal component becomes near-zero, and normalizing a near-zero vector produces an erratic direction — WASD stops translating the target and effectively causes rotation-like behavior or zero movement.

**Fix**: When the projected horizontal forward has insufficient length (near-vertical camera), fall back to computing the horizontal direction from the orbit `yaw` parameter instead, which is always well-defined. Same fix applies in the admin editor's `updateOrbitPan`.

**Changes in `PlayCanvasGsplatRuntime.ts` (`updateOrbitPan`)**:
```ts
// Replace:
const forward = cam.forward.clone();
forward.y = 0;
forward.normalize();

// With:
const forward = new Vec3();
const camForward = cam.forward.clone();
camForward.y = 0;
if (camForward.lengthSq() > 0.01) {
  forward.copy(camForward.normalize());
} else {
  // Camera nearly vertical — use yaw-derived direction
  forward.set(Math.sin(this.yaw), 0, Math.cos(this.yaw));
}
```

**Changes in `Annotation3DEditorPage.tsx` (`updateOrbitPan` inside the useEffect)**:
Same pattern — when `forward` projected to y=0 has lengthSq < 0.01, fall back to orbit-derived direction using the orbit controls' azimuthal angle. In Three.js OrbitControls, the azimuth can be read from the camera's world direction projected horizontally, or computed from the camera position relative to orbit target. Simplest: if `forward.lengthSq() < 0.01` after zeroing Y, compute from `camera.position - orbit.target` projected to horizontal.

## Fix 3: Mouse Wheel — Depth-Based Zoom Instead of Relative

**Problem**: In `PlayCanvasGsplatRuntime.ts:721-727`, orbit zoom is purely multiplicative (`distance *= 1.1 or 0.9`). This means:
- At large distances, each wheel tick moves a lot (relative to view)
- At small distances, each wheel tick moves almost nothing — zoom "stops" near the orbit target
- The zoom is relative to the orbit target center, not what the camera is actually looking at

**Fix**: Replace the pure multiplicative zoom with a **logarithmic/absolute hybrid** that guarantees perceptible movement at any distance, and bias the zoom toward the point the camera is actually looking at (the scene content in front, not just the abstract orbit center).

**Strategy**: Use a two-part zoom:
1. A multiplicative component (preserves the "zoom feel" at medium distances)
2. An additive minimum component (guarantees movement never stalls at close range)
3. When zooming in, also nudge the orbit **target** toward the camera's forward direction — this makes the zoom converge toward whatever is visually in front, not stall at an abstract grid center

**Changes in `PlayCanvasGsplatRuntime.ts` (`onWheel`)**:
```ts
private onWheel = (event: WheelEvent): void => {
  if (this.currentCameraMode === 'locked') return;
  event.preventDefault();
  const zoomIn = event.deltaY < 0;

  if (this.currentCameraMode === 'fly') {
    // Fly mode: wheel adjusts move speed (Fix #6)
    const speedFactor = zoomIn ? 1.15 : 0.85;
    this.flyMoveSpeed = Math.max(0.5, Math.min(50, this.flyMoveSpeed * speedFactor));
    return;
  }

  // Orbit mode: depth-aware zoom
  const direction = zoomIn ? -1 : 1;

  // Multiplicative component
  const scaleFactor = zoomIn ? 0.9 : 1.1;
  const newDist = this.distance * scaleFactor;

  // Additive minimum component — ensures visible movement at close range
  const minDelta = 0.05 * direction;
  const targetDist = newDist + minDelta;
  this.distance = Math.max(0.2, Math.min(200, targetDist));

  // Nudge target toward camera forward when zooming in — converges on what's in front
  if (zoomIn && this.camera) {
    const forward = this.camera.forward.clone();
    forward.y = 0;
    if (forward.lengthSq() > 0.01) {
      forward.normalize();
      const nudgeAmount = Math.min(0.02 * this.distance, 0.1);
      this.target.add(forward.mulScalar(nudgeAmount));
    }
  }

  this.updateCamera();
};
```

**Changes in `Annotation3DEditorPage.tsx`**: The admin editor uses Three.js OrbitControls which has its own wheel handling. We need to override OrbitControls' default zoom with a custom handler. The approach:
- Disable OrbitControls' built-in zoom (`orbit.enableZoom = false`)
- Add a custom wheel handler on the canvas that does the depth-aware zoom logic
- Move the orbit target forward when zooming in, similar to the PlayCanvas fix

This requires intercepting wheel events before OrbitControls and manually adjusting `orbit.target` + camera distance.

## Fix 4: Fly Mode Camera 180° Flip on Entry

**Problem**: In `PlayCanvasGsplatRuntime.ts:249-254`:
```ts
this.flyYaw = this.yaw;
this.flyPitch = this.pitch;
```
The orbit `yaw` defines the **offset** from target to camera: at yaw=0, camera is on +Z side, looking toward -Z (at the target). But the fly look formula `(sin(flyYaw)*cosP, sin(flyPitch), cos(flyYaw)*cosP)` at flyYaw=0 gives a look direction of +Z. This is the **opposite** direction — a 180° flip.

In `Annotation3DEditorPage.tsx:1175-1178`:
```ts
flyYawRef.current = 0;
flyPitchRef.current = 0;
```
This resets to zero regardless of current orientation, which is even worse.

**Fix**: Derive `flyYaw` and `flyPitch` from the **camera's actual forward direction** at the moment of transition, not from orbit parameters. This naturally handles any camera orientation.

**Changes in `PlayCanvasGsplatRuntime.ts` (`setCameraMode`)**:
```ts
if (mode === 'fly') {
  if (this.camera) {
    // Derive fly orientation from actual camera forward direction
    const forward = this.camera.forward.clone();
    this.flyYaw = Math.atan2(forward.x, forward.z);
    const horizontalLen = Math.sqrt(forward.x * forward.x + forward.z * forward.z);
    this.flyPitch = Math.atan2(forward.y, horizontalLen);
  }
  this.flyMoveSpeed = 4.0; // Reset speed on mode entry
  this.canvas.style.cursor = 'none';
  this.canvas.requestPointerLock();
}
```

**Changes in `Annotation3DEditorPage.tsx` (fly mode toggle onClick)**:
```ts
// Instead of:
flyYawRef.current = 0;
flyPitchRef.current = 0;

// Derive from camera's current forward direction:
const cam = cameraRef.current!;
const forward = new THREE.Vector3();
cam.getWorldDirection(forward);
flyYawRef.current = Math.atan2(forward.x, forward.z);
const horizontalLen = Math.sqrt(forward.x * forward.x + forward.z * forward.z);
flyPitchRef.current = Math.atan2(forward.y, horizontalLen);
```

## Fix 5: No Jumps When Switching Fly ↔ Orbit

**Problem**: Switching from fly to orbit doesn't reconstruct orbit parameters (yaw, pitch, distance, target) from the current fly camera state. The camera position and orientation jump.

**Fix**: When transitioning from fly → orbit, reconstruct orbit state from the current camera position and forward direction:
- **target** = camera position + forward * reasonable_distance
- **yaw/pitch** = derived from camera forward direction
- **distance** = chosen reasonable value (e.g., start at a distance that keeps the camera at its current position relative to the new target)

**Changes in `PlayCanvasGsplatRuntime.ts` (`setCameraMode`)**:
```ts
} else if (mode === 'orbit') {
  if (this.camera && this.currentCameraMode === 'fly') {
    // Reconstruct orbit state from current fly camera
    const pos = this.camera.getPosition().clone();
    const forward = this.camera.forward.clone();

    // Pick a target point along the forward direction
    // Use a distance proportional to scene size, capped reasonably
    const orbitDistance = Math.max(1, Math.min(20, this.distance || 4));
    this.target = pos.clone().add(forward.mulScalar(orbitDistance));

    // Derive yaw/pitch from forward direction
    this.yaw = Math.atan2(forward.x, forward.z);
    const horizontalLen = Math.sqrt(forward.x * forward.x + forward.z * forward.z);
    this.pitch = Math.atan2(forward.y, horizontalLen);
    this.distance = orbitDistance;
  }
  document.exitPointerLock();
  this.canvas.style.cursor = 'grab';
}
```

For orbit → fly, this is already handled by Fix #4 (derive flyYaw/flyPitch from camera forward), which prevents the 180° flip and ensures continuity.

**Changes in `Annotation3DEditorPage.tsx` (fly mode toggle onClick, when exiting fly)**:
```ts
// When setting flyMode to false:
document.exitPointerLock();
// Reconstruct orbit target from current camera position + forward
const cam = cameraRef.current!;
const orbit = orbitRef.current!;
const pos = cam.position.clone();
const forward = new THREE.Vector3();
cam.getWorldDirection(forward);
const orbitDist = Math.max(1, Math.min(20, pos.distanceTo(orbit.target) || 4));
orbit.target.copy(pos.clone().add(forward.multiplyScalar(orbitDist)));
orbit.update();
```

## Fix 6: Fly Mode Mousewheel = Movespeed

**Problem**: Mousewheel currently does nothing meaningful in fly mode (it modifies `this.distance` which fly mode doesn't use). The user wants wheel to adjust fly movement speed, and this speed should only apply in fly mode.

**Fix**: Add a `flyMoveSpeed` field. In fly mode, wheel adjusts this speed. WASD movement uses `flyMoveSpeed` instead of the hardcoded `4.0`. The speed is displayed or indicated somehow so the user knows their current speed.

**Changes in `PlayCanvasGsplatRuntime.ts`**:
1. Add field: `private flyMoveSpeed = 4.0;`
2. In `onWheel` (see Fix #3 code above) — when `currentCameraMode === 'fly'`, adjust `flyMoveSpeed` instead of `distance`
3. In `updateFlyCamera` — use `this.flyMoveSpeed` instead of hardcoded `speed = 4.0`:
   ```ts
   const speed = this.flyMoveSpeed;
   const effectiveSpeed = keyboard.isPressed(16) ? speed * 3 : speed;
   ```
4. In `setCameraMode` when entering fly — reset `this.flyMoveSpeed = 4.0;`

**Changes in `Annotation3DEditorPage.tsx`**:
1. Add a `flyMoveSpeedRef = useRef(4.0)` 
2. Add a wheel handler that, when in fly mode, adjusts `flyMoveSpeedRef.current`
3. In `updateFlyCamera`, use `flyMoveSpeedRef.current` instead of hardcoded `4`:
   ```ts
   const speed = k.has('shift') ? flyMoveSpeedRef.current * 3 : flyMoveSpeedRef.current;
   ```
4. The admin editor currently uses OrbitControls' built-in wheel. Need to add a custom wheel listener that checks `flyModeRef.current` and routes to speed adjustment vs orbit zoom accordingly.
