# Тестирование методом белого ящика

## Тест 1. Регистрация нового пользователя

**Действие.**  
На странице регистрации (`/signup`) в поле «Email» ввести `testuser@example.com`, в поле «Имя» ввести `Test User`, в поле «Пароль» ввести `password123`. Нажать кнопку «Sign Up».

**Ожидаемый результат.**  
Сервер создаёт запись в таблице `User` с ролью `CLIENT`, хеширует пароль через bcrypt (12 раундов) и возвращает HTTP-ответ с кодом 200. Тело ответа содержит JWT-токен и объект пользователя `{ id, email, name, role: 'CLIENT', memberships: [], capabilities: { isMasterAdmin: false, canAccessAdmin: false } }`. В ответе также установлена httpOnly-cookie `token` со сроком жизни 24 часа. Клиент перенаправляется на главную страницу (`/`).

**Фактический результат.**  
Совпадает с ожидаемым.

## Тест 2. Аутентификация администратора

**Действие.**  
На странице входа (`/login` или `/admin`) в поле «Email» ввести `admin@example.com`, в поле «Password» ввести `adminpass`. Нажать кнопку «Sign In».

**Ожидаемый результат.**  
Сервер находит пользователя по email, сверяет хеш пароля через bcrypt, формирует сериализованный объект пользователя с полем `capabilities.canAccessAdmin = true` (если роль `MASTER_ADMIN` или есть активные членства в организациях). Клиент получает объект пользователя через `onLogin(user)`, состояние `loading` сменяется с `true` на `false`, отображается интерфейс административной панели (компонент `<AppShell>` с навигацией и роутами). Если учётные данные неверны — под формой отображается блок `admin-error` с текстом ошибки.

**Фактический результат.**  
Совпадает с ожидаемым.

## Тест 3. Загрузка дашборда администратора

**Действие.**  
Зайти в админ-панель под учётной записью с правами доступа. Автоматически открывается страница дашборда (`/admin`).

**Ожидаемый результат.**  
При монтировании компонента `DashboardPage` отправляется запрос `GET /api/admin/stats`. Пока ответ не получен, отображается скелетон (`DashboardSkeleton`) — 6 карточек-заглушек с пульсирующей анимацией и заглушка таблицы. После успешного ответа скелетон скрывается, отображаются 6 карточек метрик: «Total Splats», «Published», «Processing», «Ready», «Failed», «Storage Used» с числовыми значениями. Ниже отображается таблица последних задач обработки (`recentJobs`) с колонками «Splat», «Version», «Job», «Status», «Updated». При ошибке сети под заголовком страницы отображается блок `admin-error` с сообщением об ошибке.

**Фактический результат.**  
Совпадает с ожидаемым.

## Тест 4. Поиск и фильтрация в списке сплатов

**Действие.**  
На странице «Splats» (`/admin/splats`) в поле поиска (плейсхолдер `Search splats...`) ввести часть названия сплата, например `garden`. В выпадающем списке статусов выбрать `PUBLISHED`.

**Ожидаемый результат.**  
При монтировании компонента `SplatsPage` отправляется `GET /api/admin/splats`, результат сохраняется в `useState`. Пока загружается — отображается скелетон таблицы на 6 строк. После загрузки отображается HTML-таблица с колонками «Splat», «Organization», «Status», «Asset», «Actions». При вводе текста в поле поиска срабатывает клиентская фильтрация через `useMemo`: строки таблицы обновляются в реальном времени, остаются только те, у которых `title`, `slug` или `organization.name` (в нижнем регистре) содержат подстроку `garden`. При выборе `PUBLISHED` из выпадающего списка статусов дополнительно отфильтровываются сплаты, оставляя только со статусом `PUBLISHED`. Если ни один сплат не удовлетворяет условиям, отображается заглушка `admin-empty` с текстом «No splats match your search». Фильтрация работает без дополнительных запросов к серверу.

**Фактический результат.**  
Совпадает с ожидаемым.

## Тест 5. Создание нового сплата

**Действие.**  
На странице списка сплатов нажать кнопку «New Splat». Открывается страница редактирования сплата (`/admin/splats/new`). На вкладке «Metadata» в поле «Title» ввести `Test Splat`, поле «Slug» автоматически заполняется значением `test-splat`. В поле «Description» ввести `A test splat for verification`. В выпадающем списке «Organization» выбрать первую организацию из списка. Нажать кнопку «Save».

**Ожидаемый результат.**  
При открытии страницы `isNew = true`, форма пуста, вкладка «Versions» недоступна. При вводе названия slug генерируется автоматически через `slugify()` (нижний регистр, замена пробелов на дефисы). При нажатии «Save» отправляется `POST /api/admin/splats` с JSON-телом `{ title: 'Test Splat', slug: 'test-splat', description: 'A test splat for verification', organizationId: '...' }`. Сервер создаёт запись в таблице `Splat` со статусом `DRAFT`. После успешного ответа происходит редирект на `/admin/splats/{newId}`, страница переходит в режим редактирования (`isNew = false`), становятся доступны вкладки «Versions» и «Preview». Сплат появляется в общем списке со статусом `DRAFT`.

**Фактический результат.**  
Совпадает с ожидаемым.

## Тест 6. Загрузка файла сплата и запуск обработки

**Действие.**  
На странице редактирования сплата (`/admin/splats/{id}`) перейти на вкладку «Upload» или открыть страницу загрузки (`/admin/splats/{id}/upload`). Выбрать файл `bicycle.ply` (формат `.ply`, размер ~15 МБ). Нажать кнопку «Upload».

**Ожидаемый результат.**  
Компонент `UploadPage` при выборе файла проверяет расширение (допустимые: `ply`, `sog`, `spz`, `zip`). При нажатии «Upload» файл отправляется через `POST /api/admin/splats/{id}/upload` с `Content-Type: multipart/form-data`. Сервер сохраняет оригинальный файл в MinIO (бакет `gsplat-assets`) с ключом вида `splats/{id}/versions/{versionId}/original.ply`, создаёт запись в таблице `Version` со статусом `PROCESSING` и добавляет задачу в очередь BullMQ `splat-processing`. Клиент получает ответ с объектом версии, таблица версий на вкладке «Versions» обновляется — появляется новая строка с номером версии > 0 и статусом `PROCESSING`. Прогресс-бар показывает состояние выполнения. При ошибке (неверный формат, превышение размера) отображается блок `admin-error` с сообщением от сервера.

**Фактический результат.**  
Совпадает с ожидаемым.

## Тест 7. Публикация сплата

**Действие.**  
На странице списка сплатов (`/admin/splats`) найти сплат со статусом `READY` (файл обработан, конвертирован, все задачи выполнены успешно). В строке этого сплата в колонке «Actions» нажать кнопку «Publish».

**Ожидаемый результат.**  
Отправляется `POST /api/admin/splats/{id}/publish`. Сервер меняет статус сплата с `READY` на `PUBLISHED`, устанавливает поле `publishedAt` в текущее время и назначает текущую обработанную версию как `servingVersion`. После ответа функция `fetchSplats()` перезапрашивает список, статус в строке меняется на `PUBLISHED`. Кнопка «Publish» исчезает, вместо неё появляются кнопки «Unpublish» и «View». При нажатии «View» открывается публичная страница сплата `/splats/{slug}`, где через `GET /api/splats/{slug}/manifest` отдаются URL-ы ассетов сцены, и 3D-вьювер загружает облако точек через nginx-прокси из MinIO.

**Фактический результат.**  
Совпадает с ожидаемым.

## Тест 8. Удаление сплата

**Действие.**  
На странице списка сплатов (`/admin/splats`) в строке сплата нажать кнопку «Delete». В появившемся диалоговом окне браузера с вопросом `window.confirm('Delete "{title}"? This action cannot be undone.')` нажать «OK».

**Ожидаемый результат.**  
Срабатывает `window.confirm()`. При подтверждении отправляется `DELETE /api/admin/splats/{id}`. Сервер удаляет запись сплата и связанные версии из базы данных (Prisma). После успешного ответа (HTTP 200) вызывается `fetchSplats()` — список перезагружается, удалённый сплат исчезает из таблицы. Если пользователь нажал «Отмена» в диалоге подтверждения — запрос не отправляется, сплат остаётся в списке. При ошибке (например, сплат не принадлежит организации пользователя) отображается блок `admin-error` с сообщением об ошибке.

**Фактический результат.**  
Совпадает с ожидаемым.

## Тест 9. 3D-редактор: загрузка облака точек и размещение маркера

**Действие.**  
На странице списка сплатов (`/admin/splats`) в строке сплата, у которого есть обработанный файл (`productionObjectKey` не пустой), нажать кнопку «3D Editor». Открывается страница `Annotation3DEditorPage`. После загрузки облака точек нажать кнопку «Place Marker», затем кликнуть левой кнопкой мыши в любую точку на земле (ground plane) в 3D-сцене. В появившейся форме в правой панели в поле «Name» ввести `Test Marker`. Выбрать цвет `#ef4444` (красный). Нажать кнопку «Create».

**Ожидаемый результат.**  
При загрузке страницы инициализируется Three.js-сцена: `WebGLRenderer`, `PerspectiveCamera`, `OrbitControls` (left=rotate, middle=dolly, right=pan), `AmbientLight`, `DirectionalLight`, `GridHelper`, `AxesHelper`. Через `extractGsplatPointCenters()` из `@gsplat/viewer-core` загружаются позиции вершин облака точек из файла на MinIO, строится `THREE.BufferGeometry` + `THREE.PointsMaterial` + `THREE.Points`, облако добавляется в сцену. После клика в режиме «Place Marker» на пересечении луча с ground plane создаётся временный маркер (`preview-*`): `THREE.Group` со `sphere` (красная сфера) и `sprite`-лейблом. Форма в правой панели заполняется координатами пересечения, кнопка «Create» активна. При нажатии «Create» отправляется `POST /api/admin/splats/{id}/markers` с координатами, заголовком, цветом и параметрами маркера. После успешного ответа маркер появляется в списке левой панели, временный префикс `preview-` заменяется на реальный ID из базы.

**Фактический результат.**  
Совпадает с ожидаемым.

## Тест 10. Создание и редактирование организации

**Действие.**  
На странице «Organizations» (`/admin/organizations`) в правой панели (форма создания) в поле «Name» ввести `Test Org`. Slug автоматически заполнится значением `test-org`. В поле «Description» ввести `Test organization`. В поле «Website» ввести `https://example.com`. Оставить чекбокс «Public page» включённым. Нажать кнопку «Create Organization».

**Ожидаемый результат.**  
При монтировании `OrganizationsPage` загружается список организаций через `GET /api/admin/organizations`. Пока загружается — отображается скелетон. Таблица содержит колонки: «Organization» (превью + название + slug + описание), «Members», «Splats» (published / total), «Public», «Actions». При заполнении формы slug автоматически генерируется из названия через `slugify()`. При нажатии «Create» отправляется `POST /api/admin/organizations` с JSON-телом `{ name: 'Test Org', slug: 'test-org', description: 'Test organization', websiteUrl: 'https://example.com', isPublic: true }`. Сервер создаёт запись в таблице `Organization`. После успешного ответа список перезагружается, новая организация появляется в таблице, автоматически выделяется (класс `is-selected`), форма переключается в режим редактирования — меняется заголовок на «Edit Group», появляется секция `PreviewImageEditor` для загрузки превью-изображения. При нажатии «Edit» на другой строке таблицы форма заполняется данными выбранной организации.

**Фактический результат.**  
Совпадает с ожидаемым.
